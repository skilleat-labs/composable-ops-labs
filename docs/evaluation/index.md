# 평가 개요

본 모듈의 평가는 **실습 산출물**과 **팀 제안서**로 구성됩니다. 시험이 아니라 **"일을 해냈는가"**를 확인하는 평가입니다.

---

## 평가 구성

| 항목 | 비중 | 제출 시점 |
| --- | --- | --- |
| 실습 산출물 (Phase 0–6) | 40% | Phase 종료 시 Git 커밋으로 확인 |
| 팀 제안서 및 발표 | 40% | Phase 7 |
| 개인 회고 | 20% | Phase 7 종료 후 |

---

## 실습 산출물 평가 항목

각 Phase의 결과물이 본인 GitHub 레포의 main 브랜치에 반영되어 있어야 합니다.

| Phase | 확인 항목 |
| --- | --- |
| Phase 2 | `order-api`에 Retry + Circuit Breaker 적용 코드 |
| Phase 3 | `.github/workflows/ci.yml` 파일 및 Actions 실행 성공 이력 |
| Phase 4 | `k8s/argocd-application.yaml` 및 ArgoCD가 Synced 상태인 스크린샷 |
| Phase 5 | `phase5/log_agent.py` 동작 데모 |
| Phase 6 | 팀별 Multi-Agent 코드 폴더 |

---

## 팀 제안서 평가 항목

| 항목 | 배점 | 기대하는 내용 |
| --- | --- | --- |
| 기술적 정확성 | 25% | 용어/개념 오류 없음, 실제 동작하는 코드 근거 |
| 의사결정의 근거 | 35% | "왜 이 결정을 했는가"에 대한 설득력 있는 논거 |
| 한밭푸드 맥락 반영 | 20% | 일반론이 아니라 한밭푸드의 규모/팀/예산에 맞는 제안 |
| 발표 명료성 | 20% | 3분 안에 핵심이 전달됨 |

---

## 개인 회고 평가 항목

| 항목 | 기준 |
| --- | --- |
| 솔직성 | "잘 몰랐다", "실수했다"를 있는 그대로 쓰기 |
| 구체성 | "어려웠다" (X) → "~의 ~부분이 ~때문에 어려웠다" (O) |
| 실용성 | 배운 것 중 **실제 업무에 쓸 것 하나**를 고를 수 있는가 |
